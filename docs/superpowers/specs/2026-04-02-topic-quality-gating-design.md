# Topic Quality Gating Design

## Goal
Improve script quality without losing too many candidates by adding a shared pre-persistence normalization and validation layer across all topic-writing paths.

This design keeps the current post-persistence audit worker, but makes it harder for malformed or low-value text to enter the database in the first place.

## Scope
This design covers:
- deep research worker writes
- variant expansion writes
- manual topic search / hub-driven writes
- script persistence rules for all topic text fields
- audit timing and responsibility boundaries

It does not replace the audit worker. It strengthens the layer before audit.

## Current State
The current system already has:
- a unified topic worker for discovery and audit cadence
- a separate variant expansion worker
- a separate audit worker
- a used-script flag that prevents a script from being selected again
- family-level reuse through fresh variants

Current weakness:
- weak drafts can still be persisted as `pending`
- small-tier scripts, especially `8s` and some `16s`, can be too short or awkward
- generated text can still contain dash characters
- research outputs can contain stale temporal phrasing that no longer fits the current date context
- manual search paths can drift from worker paths unless they share the same gate

## Design Principles
1. Normalize early, reject late.
2. Keep the audit worker as the post-persistence quality arbiter.
3. Make manual search and worker generation use the same validation logic.
4. Prefer repair over loss when a draft can be cleaned safely.
5. Apply the same text hygiene rules to every persisted topic text field.

## Architecture

### Layer 1: Shared pre-persistence quality gate
A deterministic shared helper validates candidate text before persistence.

Responsibilities:
- strip disallowed dash characters from all topic text fields
- enforce tier-specific minimum length, especially for `8s`
- reject malformed or truncated text before it enters the database
- normalize outdated temporal phrasing when the current year makes the original wording stale
- ensure the same rule set is used by worker generation and manual search

This layer is not the audit agent.
It is a fast, local, deterministic gate that runs before `topic_scripts` rows are inserted or updated.

### Layer 2: Post-persistence audit worker
The existing audit worker remains responsible for:
- reading `pending` scripts
- applying deterministic checks
- applying Gemini-based quality scoring
- writing `pass`, `needs_repair`, or `reject`
- promoting a family to `active` when pass coverage exists

This layer remains the source of truth for final script quality.

### Layer 3: Selection and immutability
Selection stays strict:
- only `pass` scripts are eligible
- used scripts are never eligible again
- family reuse happens only through fresh unused variants
- used scripts are historical records, not deletion targets

## Validation Rules

### No-dash rule
All persisted topic text fields must be normalized to remove dash separator characters such as em-dashes and similar long-dash punctuation.

This rule targets separator-style dashes, not ordinary hyphenated compounds that are part of a normal word or product name.

This applies to:
- titles
- rotations
- CTAs
- scripts
- source summaries
- disclaimers
- research captions
- any other persisted topic text field generated by the topic pipeline

Preferred behavior:
- remove the dash when the sentence remains natural
- if the sentence becomes awkward, rewrite once
- reject only if the text cannot be repaired without losing meaning

Rationale:
- the system should not generate bad text and then punish it later
- the cleanup should be cheap and automatic where possible

### Stronger `8s` scripts
`8s` scripts are currently too short.

New rule:
- allow a small word-count increase for `8s` content
- keep the result compact, but not clipped
- require at least one extra meaningful phrase when the script is too terse

Target behavior:
- `8s` should read like a complete spoken hook, not a fragment
- the final output should still fit the tier, but with slightly more usable content than today

### Tier-aware quality gate
The shared gate should be stricter for smaller tiers:
- `8s`: strictest length and completeness checks
- `16s`: balanced checks for clarity and sentence arc
- `32s`: allow fuller context, but still enforce clean spoken structure

### Time-aware research normalization
Deep research output must be interpreted against the current date context.

Examples:
- statements like "starting 2025" may be stale in 2026
- dated phrasing should be rewritten or rejected when it is no longer contextually correct

The goal is to avoid storing research that is already out of date.

## Entry-Point Behavior

### Deep research worker
Before inserting any candidate text:
- run the shared pre-persistence gate
- normalize dash characters
- repair short `8s` outputs if needed
- reject stale temporal phrasing if it cannot be corrected

### Variant expansion worker
Before persisting a variant:
- run the same shared gate
- do not allow a looser path than the deep research worker
- keep the output consistent with the main research path

### Manual topic search / hub-driven write paths
Manual search must call the exact same helper.

Reason:
- if the worker path is hardened but manual search is not, the database will drift again
- manual writes should not become a bypass route for low-quality text

## Failure Policy
1. Try to normalize.
2. Try one rewrite when the text can be made natural.
3. If the candidate still fails, do not persist it.
4. Let audit handle final quality only for persisted candidates.

This keeps the system from inserting obvious garbage while avoiding unnecessary loss.

## Expected Outcome
After this change:
- fewer malformed scripts enter the database
- `8s` scripts become more usable
- dash characters stop appearing in persisted topic text
- time-sensitive research becomes less stale
- worker paths and manual paths behave consistently
- audit backlog should shrink because fewer weak drafts reach `pending`

## Testing Plan

### Unit coverage
- no-dash normalization across topic text fields
- `8s` script minimum-length repair
- time-aware research normalization
- manual search and worker writes use the same shared helper

### Integration coverage
- research worker persists cleaned scripts only
- variant expansion persists cleaned scripts only
- manual search cannot bypass the shared gate
- audit still receives only persisted candidates and does not become the pre-persistence gate

### Regression coverage
- used scripts remain non-selectable
- used scripts remain non-deletable
- family-level reuse still works through fresh variants

## Success Criteria
- fewer `needs_repair` and `reject` rows caused by simple formatting or length problems
- no persisted topic text contains dash characters
- `8s` drafts are measurably less clipped
- manual search and worker output follow the same policy
- audit remains post-persistence, not pre-persistence
