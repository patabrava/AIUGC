# Deep Research Flow

1. **Trigger**: `app/features/topics/handlers.py` seeds batches that request scripts. It calls `list_topic_suggestions()` first; if the bank is short for the requested tier, it performs one canonical warm-up run that researches 3 unique value/product seeds and repopulates `topic_scripts` before re-querying suggestions.
2. **Prompt construction**: `prompt1_research.txt` is assembled via `build_prompt1()` in `app/features/topics/prompts.py`. That builder also extracts the frozen topic seeds and hook bank from `prompt1_8s.txt`, so the research prompt receives a curated topic list and a hook-bank context.
3. **Dossier normalization**: `normalize_topic_research_dossier()` now parses Gemini Deep Research prose locally; it keeps JSON parsing only as a local compatibility fallback and never asks Gemini to normalize into JSON for the value warm-up path.
4. **Lane harvesting**: Each accepted lane is turned into a lane-specific dossier context, then `generate_topic_script_candidate()` emits text-only PROMPT_1 stage 3 output and normalizes the script locally for `8s`, `16s`, and `32s`.
5. **Persistence**: `topic_scripts` rows now store canonical base-tier coverage with `bucket="canonical"` plus `framework`, `hook_style`, `lane_key`, `target_length_tier`, and `topic_research_dossier_id`, while later expansion jobs are reserved for non-canonical variants.
6. **Fallback and validation**: Script outputs must still pass the tier envelopes and German-only checks before persistence; identical normalized script text is skipped at the database boundary instead of being cloned into other buckets, and duplicate tier collisions must never flatten canonical coverage.
7. **Future reuse**: Next time a batch asks for the same tier/topic, the handler reuses stored bank suggestions first and only warms the bank again if the canonical coverage is missing.
