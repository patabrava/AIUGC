# AGENTS.md

0) Non-negotiable Locality Envelope (default)
- Vanilla-first: prefer platform primitives; default to vanilla first, and only use frameworks if justified.
- Dependency budget: 0 by default; max 1-2 per concern, only if justified.
- File budget: Fewer is better. Create as fewer files as possible to achieve production-grade quality without creating context overwhelm (zero unnecessary files).
- LOC budget: target <= 1000 LOC/file; split when > 2000 LOC. Avoid context rot growth.
- Every Plan offered must explicitly state: {files, LOC/file, deps}.

1) Role routing (global)
- Keep bridgecode playbooks in `bridgecode/` and generated artifacts in `agents/`; do not mix the two folders.
- When doing backend development always follow the LLM_FRIENDLY_ENGINEERING_BACKEND instructions.
- When doing frontend design always follow the LLM_FRIENDLY_ENGINEERING_FRONTEND instructions.
- When planning or debugging always follow the LLM_FRIENDLY_PLAN_TEST_DEBUG instructions.
- Whenever starting a conversation (no previous turns detected) check only one time, to start the conversation, the `agents/canon.md` file to get an overview of the repo you're working with.
  - If `agents/canon.md` does not exist, instruct the user to use `LIRA` and the sub-role `senior`.
- Whenever `EYE` fixes an error, through simple implementation or isolated debugging (with or without loop-breaker), add a compoacted, dense one-line with the solution to the error (to be prevented) in the Specific repo rules within the `/AGENTS.md` file.
  - If a correction for an error already exist, either modify that one-line (if the correction was insufficient) or add extra instructions (if new related errors emerged but the initial correction was correct).
- Codex bridgecode routing:
  - If the user invokes `INSTRUCT`, immediately read `bridgecode/INSTRUCT.md` and follow it. When that playbook routes to negentropy behavior, read `bridgecode/negentropy.md`.
  - If the user invokes `LIRA` for a new project or architecture definition, read `bridgecode/LIRA.md` and `bridgecode/architect.md`.
  - If the user invokes `LIRA` for an existing codebase review or asks to audit/review the repo, read `bridgecode/LIRA.md` and `bridgecode/senior.md`.
  - If the user invokes `EYE`, or asks to implement/debug work using bridgecode execution, read `bridgecode/EYE.md` and `bridgecode/plan-code-debug.md`.
- `bridgecode/` contains the Codex playbooks. `agents/` is reserved for generated artifacts such as `canon.md`, `plan.md`, `review.md`, `negentropized_instructions.md`, and `agents/testscripts/failure_report.md`.

2) Specific repo rules
- Gemini Interactions accepts `background` Deep Research requests but its follow-up `responseSchema` rejects OpenAI-style `additionalProperties`; strip unsupported schema fields before Gemini JSON generation and log interaction-id/status transitions until completion.
- Async batch background tasks must not run blocking Gemini poll loops on the request event loop; offload long Deep Research waits, and always translate VEO HTTP 429/quota responses into structured `RATE_LIMIT` errors instead of generic 500s.
- In `S2_SEEDED`, treat script review as per-post state in `seed_data`: any script edit resets review to pending, batch advance must require every post marked approved or removed plus at least one approved, and removed posts must be excluded from later prompt/video transitions.
- For PROMPT_1 Deep Research, request the exact first-pass topic count in one interaction per researchable post type, retry transient Gemini poll 429/5xx errors with bounded backoff, and never restart a finished research run just because the downstream normalization/validator step needs repair.
- Topic Deep Research for German audiences must enforce German-only output twice: prompt-level in `prompt1.yaml` and validation-level in `app/features/topics/agents.py`, rejecting and retrying any English or mixed-language `topic/script/source_summary/tone/disclaimer` before persistence.
- PROMPT_1 normalization must defensively default malformed or missing item fields like `framework` before validation so a completed Deep Research batch never dies before `create_post_for_batch(...)` and silently leaves `posts` empty.
- VEO portrait runs must send REST `parameters.aspectRatio` and `parameters.resolution` explicitly and must not locally reject valid Veo 3.1 `9:16` + `1080p` requests.
- Veo extended-duration jobs must never reuse the legacy `submitted/processing` queue lane; store them under dedicated extended statuses until the chain reaches its final combined output or an old single-clip worker can finalize the 4-second base segment before any extension hop runs.
- Veo extended-chain workers must treat Google `429 RESOURCE_EXHAUSTED` on extension hops as a cooldown on the same post, persisting `next_retry_at` and staying in `extended_processing`; failing immediately strands a paid base render and wastes the resumable chain.
- VEO talking-head prompts must make the speech contract explicit in both base and extension prompts: require German-only verbatim delivery of the approved `seed_data.script`, forbid translation/paraphrase/ad-lib, and reserve the closing hold for silence or Veo will drift to English and ignore the approved line.
- VEO talking-head prompts must include an explicit post-dialogue `Ending` instruction that stops speech after the final word and keeps the mouth at rest; generic hold-smile action text alone is insufficient and causes false continuation cuts near clip end.
- VEO talking-head batches must not target full-clip speech: keep upstream scripts around 5-6 spoken seconds inside the current 8-second flow so there is real budget for the end hold and room-tone decay.
- Video submission prompts must collapse all audio guidance into one qualitative `Audio:` block with natural room-tone decay; never send dead-silence requests, RT60/dBFS/LUFS-style metrics, or duplicate audio/post/SFX sections, and Veo exclusions must ship via REST `parameters.negativePrompt` rather than inline `no...` negative text in the main prompt.
- PROMPT_2 hook expansions must update `parse_prompt2_response()` in lockstep: recognize every allowed hook-family prefix and split consecutive one-line scripts even without blank separators, or adjacent scripts will merge and corrupt dialog extraction.
- Live seeding feeds must not rely only on pre/post Deep Research stage wrappers; thread Gemini interaction polling callbacks through `generate_gemini_deep_research()` into topic discovery so the feed emits heartbeat/progress events during long research waits instead of appearing live-but-stale.
- Gemini Deep Research live feeds must consume the provider stream with `stream=True` plus `thinking_summaries="auto"` and resume from `last_event_id`; pure status polling only produces repetitive synthetic updates and cannot show real research progress.
- Gemini Deep Research SSE does not emit a constant thought stream: handle `interaction.status_update`, use a short idle read-timeout on stream calls, and reopen `interactions.get(..., stream=True)` continuously until completion or the UI will freeze behind a long-lived silent socket.
- Gemini Deep Research stream resumes must cap consecutive idle read timeouts and fall back to `_generate_gemini_deep_research_poll_result(...)`; otherwise value/topic runs can stall for minutes behind a silent SSE socket even though the polling path could finish the interaction.
- Gemini Deep Research poll fallback must retry bounded `httpx.TransportError` failures before aborting the run; transient TLS/connect drops during fallback polling are recoverable and must not kill value/topic generation immediately.
- S2 script review must size the editable textarea from the actual script length instead of hard-coding `rows="3"`; otherwise longer lifestyle scripts look truncated in the UI even when `seed_data.script` is stored intact.
- Batch detail views after `S2_SEEDED` must filter removed posts out of the rendered post list and all stage counters; once a script is rejected, it should disappear from prompt/video/QA/publish screens instead of lingering as excluded noise.
- Batch detail HTMX polling must never `outerHTML`-swap the root while an embedded `<video>` is actively playing; guard refreshes on playback state or the native player will reset a valid asset mid-watch and look like a broken short-loop playback failure.
- `S5_PROMPTS_BUILT -> S6_QA` must count only active posts in the worker transition gate; if removed or `video_excluded` posts remain in the batch-level total, QA will never unlock even though all eligible videos finished.
- `S5_PROMPTS_BUILT -> S6_QA` cannot rely only on the single post-completion edge; the worker must reconcile stuck `S5` batches every poll cycle or any one-time missed transition leaves fully completed batches stranded in prompts forever.
- Lifestyle topic generation must run through the same `deduplicate_topics(..., threshold=0.35)` gate against `topic_registry` plus already-collected batch topics before `add_topic_to_registry(...)`/`create_post_for_batch(...)`; exact DB uniqueness alone only stops byte-identical triples and lets repetitive lifestyle titles recur across batches.
- PROMPT_2 lifestyle scripts are often single-sentence lines, so `generate_lifestyle_topics()` must apply the same empty-rotation fallback as `convert_research_item_to_topic()`: if CTA stripping would blank the script, keep the full sentence as `rotation` and derive only a short trailing CTA fragment.
- VEO extended prompt segmentation must treat digit-dot-digit (e.g., `4.180`) as non-boundary; sentence splitting on numeric punctuation chops German scripts mid-number and causes segment replay.
- VEO MP4 duration verification must read mvhd version-0 offsets correctly (timescale at +16, duration at +20); an off-by-4-byte parser will read zero timescale from valid Google MP4s and falsely fail completed videos as duration mismatches.
- VEO dry-run previews must emit every sentence hop, not just the base clip and first extension; otherwise long scripts appear truncated even when the worker chain can advance through the remaining segments.
- PROMPT_1 trimming must preserve complete sentences only; never pop words from the tail of a valid German script because that can persist a chopped fragment into `seed_data.script`, then into `prompt_builder.py` and the Veo extension chain.
- VEO extension chaining must build each new hop from the next segment index, not the stale current one, and the live worker must reuse the same structured `build_veo_prompt_segment(...)` contract as the dry-run preview; otherwise paid runs replay earlier lines, skip the last sentence, and invalidate preflight approval.
